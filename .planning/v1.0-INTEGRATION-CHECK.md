## E2E Flow Trace Results

### Flow 1: Task Capture

`InputBubble` → `POST /api/parse` → optimistic store add → `POST /api/tasks` → `replaceTask` → scene render

- **InputBubble.tsx** calls `fetch('/api/parse', { method: 'POST' })` with `{ input: trimmed }`.
- `/api/parse/route.ts` exports `ParsedTask` type and `POST` handler; `InputBubble` imports `ParsedTask` from `@/app/api/parse/route`.
- On parse success: `store.getState().addTask(tempTask)` adds optimistic entry; `newTaskIds` set updated so `useIsNewTask` returns true for entrance animation.
- Background `fetch('/api/tasks', { method: 'POST' })` fires; on success `store.getState().replaceTask(tempId, real)` swaps temp ID for real DB ID.
- On failure: `store.getState().removeTask(tempId)` rolls back.
- `useTasksWithHorizon` in `HorizonScene` → `TaskNodes` reactively re-renders `TaskNode` (card or sprite) for the new task.
- Camera pan to horizon via `cameraStore.setState({ targetZ, isAnimating: true })` using `getZDepth` from `@/lib/horizons`.
- **Status: COMPLETE**

### Flow 2: Task Completion

`TaskCard`/`TaskSprite` click → `selectTask` → `TaskDetail` opens → Complete button → `startCompletion` → `CompletionBurst` → `PATCH /api/tasks/[id]` → `finishCompletion` removes from store → no reappear on refresh

- `TaskCard` and `TaskSprite` both call `store?.getState().selectTask(task.id)` on click.
- `TaskDetail` uses `useSelectedTask()` which reads `selectedTaskId` from store; panel renders when task is non-null.
- `handleComplete` calls `state.startCompletion(task.id)`, setting `completingTaskIds`. `useIsCompleting` in `TaskCard`/`TaskSprite` reacts, triggering CSS dissolution/R3F scale-down animations.
- `HorizonScene`'s `TaskNodes` subscribes to `completingTaskIds` changes, spawns `CompletionBurst` at task position, then calls `store.getState().finishCompletion(b.id)` in `onComplete`.
- `finishCompletion` removes the task from `state.tasks` — task disappears from scene.
- 4 seconds after Complete click, `PATCH /api/tasks/[id]` fires with `{ status: 'completed' }`. The `UPDATABLE_FIELDS` allowlist in the PATCH route includes `status` — update succeeds.
- On next page load, `page.tsx` queries `WHERE status = 'active'` so completed task does not reappear.
- Undo path: `cancelCompletion` + `restoreTask` cancel the timeout before PATCH fires, bringing task back.
- **Status: COMPLETE**

### Flow 3: Drift Accountability

Page load → RSC drift increment → `DriftNotification` toast → task with 3+ drifts → `TaskDetail` shows drift prompt → Recommit/Snooze/Drop actions

- `page.tsx` runs a DB UPDATE for active tasks whose `targetDateLatest < now`, incrementing `driftCount` and advancing the date window. Returns `drifted` rows.
- Tasks newly reaching `driftCount >= 3` get `needsRefinement: true` set in a second UPDATE.
- `driftSummary` prop (`{ count: drifted.length }`) passed: `page.tsx` → `SceneLoader` → `HorizonScene`.
- `HorizonScene` renders `<DriftNotification count={driftSummary.count} />` when count > 0.
- `DriftNotification` self-dismisses after 5s via internal timers.
- In `TaskDetail`, when `task.driftCount >= 3 && actionResult === null`, a drift accountability section renders with Recommit/Snooze/Drop buttons. Recommit calls `handleReschedule(task.horizon)` (same horizon, resets dates). Snooze calls `handleReschedule('someday')`. Drop calls `handleDrop()`.
- All three ultimately call `PATCH /api/tasks/[id]` (reschedule patches dates; drop patches status).
- **Status: COMPLETE**

### Flow 4: Refinement

`needsRefinement=true` task → `TaskDetail` auto-fetches `POST /api/refine` → shows clarifying question → user responds → `POST /api/refine` with `userResponse` → store update + DB write → title updated, flag cleared

- `TaskDetail` `useEffect` fires when `task.needsRefinement` is true and `task.refinementPrompt` is null. Calls `fetch('/api/refine', { method: 'POST', body: { taskId } })`.
- `/api/refine/route.ts` MODE 1 (no `userResponse`): fetches task from DB, calls Claude Haiku, returns `{ clarifyingQuestion, suggestedTitle }`, writes JSON to `refinementPrompt` column.
- `TaskDetail` stores result in `refinementData` state and also calls `store.getState().updateTask(task.id, { refinementPrompt: JSON.stringify(data) })` so next open skips the API call and parses from the stored prompt.
- User types response and clicks Refine. `handleRefinementSubmit` calls `fetch('/api/refine', { method: 'POST', body: { taskId, userResponse } })`.
- `/api/refine/route.ts` MODE 2: rewrites title + rawInput via Claude, writes to DB with `needsRefinement: false, refinementPrompt: null`, returns `{ title, rawInput, needsRefinement: false }`.
- `TaskDetail` calls `store.getState().updateTask(task.id, { title, rawInput, needsRefinement: false, refinementPrompt: null })` and updates local title state. Scene re-renders `TaskCard`/`TaskSprite` with new title.
- **One detail worth noting:** `/api/refine` MODE 2 response body does not include `refinementPrompt: null` explicitly (returns `{ title, rawInput, needsRefinement: false }`). However, `TaskDetail` explicitly sets `refinementPrompt: null` in the store update regardless of what the API returns (line 339), so the store is correctly cleared. The DB is also cleared via the SQL UPDATE. No break here.
- **Status: COMPLETE**

### Flow 5: List View

L key or button → Canvas hidden → `ListView` shows grouped tasks → filters work → quick complete/drop/reschedule → `TaskDetail` opens on row click

- `HorizonScene` listens for `keydown` on `l`/`L` (when target is not an input/textarea) and calls `toggleListView()`.
- `showListView` from `useTaskStore` controls: Canvas wrapper gets `display: none` (not unmounted — store, WebGL context, and R3F state all preserved), and `{showListView && <ListView />}` renders.
- `ListView` uses `useTasksWithHorizon()` (same store data) and `useTaskStore` for filter state and `setListFilter`. Grouping and filtering all computed client-side via `useMemo`.
- Quick complete: `handleQuickComplete` calls `startCompletion` → `PATCH` → `finishCompletion`. This removes the task from the store, which removes it from the list immediately.
- Quick drop: `handleQuickDrop` calls `startDrop` → `PATCH` → `finishDrop` (after 300ms).
- Quick reschedule: `handleQuickReschedule` calls `updateTask` optimistically → `PATCH` with new dates.
- Row click: `store?.getState().selectTask(task.id)` → `TaskDetail` (mounted in `HorizonScene`, always present regardless of list/canvas mode) renders with the selected task.
- `TaskDetail` uses `useSelectedTask()` which works from the shared store. TaskDetail is rendered outside the Canvas div, at the `HorizonScene` level, so it is available in both 3D and list view.
- **Status: COMPLETE**

## Cross-Phase Prop Chain Verification

### SceneLoader → HorizonScene → TaskNodes → TaskNode → TaskCard/TaskSprite

```
page.tsx (RSC)
  → SceneLoader initialTasks={allTasks} driftSummary={...}      [confirmed]
    → TaskStoreProvider initialTasks={initialTasks}              [confirmed]
      → HorizonScene (dynamic import, ssr:false) driftSummary   [confirmed]
        → TaskNodes (internal component)
          → useTasksWithHorizon()                               [confirmed]
          → getTaskPosition + applyOverlapAvoidance             [confirmed]
          → TaskNode task={task} position={[x,y,z]}            [confirmed]
            → TaskCard task position isNew (when isCard)        [confirmed]
            → TaskSprite task position isNew (when !isCard)     [confirmed]
```

All prop types match: `Task` (with `horizon`) flows into `TaskNode`; `TaskRow` fields like `driftCount`, `needsRefinement`, `hardDeadline` used in both `TaskCard` and `TaskSprite` for visual indicators.

### TaskStoreContext availability

`TaskStoreProvider` wraps `HorizonScene` in `SceneLoader`. All client components that use `useContext(TaskStoreContext)` directly (`InputBubble`, `TaskDetail`, `ListView`, `TaskCard`, `TaskSprite`, `HorizonScene`'s `TaskNodes`) are rendered inside this provider. `HorizonScene` uses `useTaskStore` (which calls `useContext` internally) — also inside the provider boundary. No component attempts to use the store outside the provider.

### driftSummary prop chain

`page.tsx` computes `drifted.length > 0 ? { count: drifted.length } : null` and passes it as `driftSummary` to `SceneLoader`. `SceneLoader` passes it to `HorizonScene`. `HorizonScene` renders `<DriftNotification count={driftSummary.count} />` only when `driftSummary && driftSummary.count > 0`. Types match throughout (`{ count: number } | null | undefined`). Error path (`<SceneLoader initialTasks={[]} error />`) omits `driftSummary`, which is fine — `HorizonScene` is not rendered in the error branch.

### InputBubble → /api/parse → store.addTask → R3F scene update

Verified in Flow 1 trace. The `SceneInvalidator` component subscribes to the Zustand vanilla store via `store.subscribe(() => invalidate())` inside the Canvas. Any store mutation (including `addTask`) triggers `invalidate()`, causing R3F to re-render on the next frame. `frameloop="demand"` means R3F only renders when explicitly invalidated, so this subscription is the critical link between store changes and scene updates.

### TaskDetail → /api/tasks PATCH → store.updateTask → scene re-render

All PATCH calls in `TaskDetail` (`saveTitle`, `saveRawInput`, `handleReschedule`) call `store.getState().updateTask()` either before or on success, keeping the store in sync. `SceneInvalidator` subscription ensures the R3F scene re-renders. `useTasksWithHorizon` recomputes horizon for the updated task on next store read.

### HorizonScene → ListView toggle

`showListView = useTaskStore((s) => s.showListView)` and `toggleListView = useTaskStore((s) => s.toggleListView)` are used directly in `HorizonScene`. The `<ListView />` conditional and the `display: showListView ? 'none' : 'contents'` Canvas wrapper are both driven by this single store value.
