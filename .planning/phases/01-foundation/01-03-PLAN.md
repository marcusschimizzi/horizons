<feature>
  <name>Horizon Math: getHorizon, getZDepth, getZDepthRange</name>
  <files>src/lib/horizons.ts, src/lib/__tests__/horizons.test.ts</files>
  <behavior>
    `getHorizon(targetDate, now)` maps a DateRange + current time to one of six horizons:
    - No targetDate -> 'someday'
    - Midpoint <= 1 day from now -> 'immediate'
    - Midpoint <= 7 days from now -> 'this-week'
    - Midpoint <= 30 days from now -> 'this-month'
    - Midpoint <= 90 days from now -> 'this-quarter'
    - Midpoint <= 365 days from now -> 'this-year'
    - Midpoint > 365 days from now -> 'someday'
    - Past dates (negative daysOut) -> 'immediate' (overdue tasks show up close)

    `getZDepth(horizon)` returns the midpoint Z of each horizon band:
    - 'immediate' -> -2.5 (midpoint of 0 to -5)
    - 'this-week' -> -10 (midpoint of -5 to -15)
    - 'this-month' -> -22.5
    - 'this-quarter' -> -40
    - 'this-year' -> -65
    - 'someday' -> -100

    `getZDepthRange(horizon)` returns { zMin, zMax }:
    - 'immediate' -> { zMin: 0, zMax: -5 }
    - 'someday' -> { zMin: -80, zMax: -120 }

    Test cases:
    - getHorizon(undefined) === 'someday'
    - getHorizon({earliest: tomorrow, latest: tomorrow}, now) === 'immediate'
    - getHorizon({earliest: now + 3 days, latest: now + 5 days}, now) === 'this-week'
    - getHorizon({earliest: now + 14 days, latest: now + 20 days}, now) === 'this-month'
    - getHorizon({earliest: now + 60 days, latest: now + 70 days}, now) === 'this-quarter'
    - getHorizon({earliest: now + 200 days, latest: now + 250 days}, now) === 'this-year'
    - getHorizon({earliest: now + 400 days, latest: now + 500 days}, now) === 'someday'
    - getHorizon({earliest: yesterday, latest: yesterday}, now) === 'immediate' (overdue)
    - getZDepth('immediate') === -2.5
    - getZDepth('someday') === -100
    - All six horizons have correct midpoints
    - getZDepthRange('this-week') === { zMin: -5, zMax: -15 }
  </behavior>
  <implementation>
    Implement exactly as specified in the research code examples in `01-RESEARCH.md`. The `Horizon` type, `HORIZON_BANDS` array, `getHorizon()`, `getZDepth()`, and `getZDepthRange()` functions.

    One addition beyond research: handle past dates (negative daysOut) by clamping to 'immediate'. This handles overdue tasks naturally — they show up closest to the user.

    Export the `Horizon` type, `HorizonBand` interface, `HORIZON_BANDS` constant, and all three functions.
  </implementation>
</feature>

<feature>
  <name>Spatial Positioning: getTaskPosition and applyOverlapAvoidance</name>
  <files>src/lib/spatial.ts, src/lib/__tests__/spatial.test.ts</files>
  <behavior>
    `getTaskPosition(taskId, horizon)` returns deterministic { x, y, z } for a task:
    - Same taskId + horizon always produces same x/y/z (deterministic via hash)
    - Z is within the horizon's band range (e.g. 'this-week' Z between -5 and -15)
    - X spread is wider for distant horizons (immediate: +-4, someday: +-22)
    - Y spread is taller for distant horizons (immediate: +-2, someday: +-7)

    `applyOverlapAvoidance(positions, minDistance, iterations)` nudges overlapping tasks apart:
    - Only compares tasks within the same horizon band
    - After avoidance, overlapping pairs are further apart than before
    - Does not move tasks that are already separated
    - Is idempotent-ish (multiple calls converge rather than diverge)

    Test cases:
    - getTaskPosition('abc', 'immediate') === getTaskPosition('abc', 'immediate') (deterministic)
    - getTaskPosition('abc', 'immediate') !== getTaskPosition('xyz', 'immediate') (different IDs produce different positions)
    - getTaskPosition('abc', 'immediate').z is between 0 and -5 (within immediate band)
    - getTaskPosition('abc', 'this-week').z is between -5 and -15
    - getTaskPosition('abc', 'someday').z is between -80 and -120
    - |getTaskPosition('abc', 'immediate').x| <= 4 (within immediate xSpread)
    - |getTaskPosition('abc', 'someday').x| <= 22 (within someday xSpread)
    - applyOverlapAvoidance with two tasks at the same position -> they move apart
    - applyOverlapAvoidance with tasks already separated -> positions barely change
    - applyOverlapAvoidance returns a Map with entries for all input task IDs
  </behavior>
  <implementation>
    Implement using the cyrb53 hash + mulberry32 PRNG from the research code. This gives deterministic pseudo-random positions per task ID.

    The SPREAD_CONFIG constant defines x/y spread per horizon. Wider spread for distant bands reinforces the perspective vanishing point effect — close tasks are clustered tight, distant tasks fan out.

    `applyOverlapAvoidance` does a simple N^2 pair comparison within same-horizon groups, nudging apart tasks closer than `minDistance`. Run for `iterations` passes (default 3). This is best-effort, not guaranteed, which is fine for 50-200 tasks.

    Export `TaskPosition` interface, `SPREAD_CONFIG` constant, `getTaskPosition()`, and `applyOverlapAvoidance()`.
  </implementation>
</feature>
