# Milestone v1.0: Spatial MVP

**Status:** ✅ SHIPPED 2026-02-28
**Phases:** 1–7
**Total Plans:** 21

## Overview

Horizon is a personal-use PoC that bets a single thesis: a 3D spatial view where Z-depth equals time produces a meaningfully different relationship with task management than any flat list. The build follows a strict dependency chain — foundation and data infrastructure first, then the 3D scene (the thesis), then camera, capture, mutation, and polish.

## Phases

### Phase 1: Foundation

**Goal:** The project structure exists, Postgres is connected, Drizzle schema and migrations run, all shared types are defined, and horizon math (`lib/horizons.ts`) is implemented — every subsequent phase builds on this without touching setup again.
**Depends on:** Nothing (first phase)
**Plans:** 3 plans

Plans:
- [x] 01-01: Next.js 15 scaffold, R3F 9 version matrix, Tailwind v4, Docker Compose Postgres (Wave 1)
- [x] 01-02: Drizzle schema, globalThis connection singleton, migrations (Wave 2)
- [x] 01-03: Shared types, horizon math, spatial scatter algorithm with TDD (Wave 2)

**Details:**
- Next.js 16.1.6 App Router, R3F 9.5.0, Drizzle 0.45.1, Zustand 5, Railway Postgres
- six horizon bands: immediate → someday, mapped to Z-depth
- `getHorizon(targetDate, now)` and `getZDepth(horizon)` implemented with TDD
- globalThis singleton pattern for DB pool to prevent HMR exhaustion
- Canvas must use `dynamic(ssr: false)` — not just `"use client"` — or the build crashes

### Phase 2: Data Layer

**Goal:** The app has real task data in Postgres, a working CRUD API, and a Zustand store that hydrates from the server — the 3D scene has everything it needs to render without building any UI yet.
**Depends on:** Phase 1
**Plans:** 2 plans

Plans:
- [x] 02-01: CRUD API routes (GET/POST/PATCH/DELETE) + seed script with ~35 representative tasks (Wave 1)
- [x] 02-02: Zustand vanilla store + SceneLoader client boundary + page.tsx server data fetch (Wave 1)

**Details:**

### Phase 4: Camera

**Goal:** The signature interaction works — users can fly through time by scrolling, feel momentum and easing as they move, cannot scroll out of bounds, can snap back to the present instantly, and experience subtle parallax.
**Depends on:** Phase 3
**Plans:** 2 plans

Plans:
- [x] 04-01: Camera store + CameraRig: scroll-to-Z, damp3 lerp in useFrame, iOS spring-back + fog-stop boundaries (Wave 1)
- [x] 04-02: SnapToPresent button + Home key shortcut, mouse-follow depth parallax (Wave 2)

**Details:**
- Vanilla Zustand store (createStore) for camera state — non-reactive getState() in useFrame
- Exponential rubber-band formula for near boundary overscroll
- Spring-back triggers when velocity < 0.5 while past nearBoundary
- Combined damp3 on [parallaxX, parallaxY, targetZ] — single call handles both
- DOM-to-Canvas invalidation bridge via cameraStore.subscribe in CameraRig

**Goal:** The capture loop is end-to-end — a user can type a natural language intention into the input bubble, have Haiku parse and place it in the 3D scene at the correct horizon, and see it persist after a page refresh.
**Depends on:** Phase 4
**Plans:** 3 plans

Plans:

- [x] 05-02: /api/parse route: Haiku structured extraction with zodOutputFormat, server-side key enforcement (Wave 1)

**Details:**

### Phase 6: Task Interactions

**Goal:** The mutation loop is complete — users can click any task to open its detail panel and from there complete it with a satisfying dissolution, drop it without ceremony, reschedule it to a new horizon, or edit its title and details.
**Depends on:** Phase 5
**Plans:** 3 plans

Plans:
- [x] 06-01: Store extensions, click handlers on TaskCard/TaskSprite, TaskDetail slide-in panel (Wave 1)
- [x] 06-02: Complete action with particle burst dissolution + undo toast, Drop action with instant removal (Wave 2)
- [x] 06-03: Reschedule via horizon band selector with smooth drift animation, inline title/details editing with auto-save (Wave 3)

**Details:**
- Panel always in DOM with CSS transform toggle for smooth slide-in (no mount/unmount)
- 4-second undo window before completion PATCH persists to server
- Drop has no undo — immediate PATCH, instant shrink, clinical feel
- CompletionBurst uses delta-based timing for R3F demand frameloop compatibility
- Dirty refs prevent spurious PATCH on panel open — auto-save only fires when user has edited
- Debounced auto-save: 1s debounce + immediate save on blur
- Camera auto-pans to rescheduled horizon when >20 z-units away from current view

### Phase 7: Polish

**Goal:** The experience is complete — drift accountability is tracked and surfaced, AI-powered refinement prompts work end-to-end, the list view escape hatch is fully functional, and bloom post-processing is tuned so the scene looks right with real task data.
**Depends on:** Phase 6
**Plans:** 4 plans

Plans:
- [x] 07-01: RSC drift recalculation, double-count prevention, DriftNotification toast, adaptive fog (Wave 1)
- [x] 07-02: TaskCard drift badge + needsRefinement pulse, TaskSprite drift opacity + refinement/deadline rings (Wave 1)
- [x] 07-03: /api/refine route, drift accountability prompt in TaskDetail, refinement UI (Wave 2)
- [x] 07-04: ListView with toggle, horizon grouping, filters, quick actions (Wave 2)

**Details:**
- RSC on-load drift check: page.tsx increments driftCount, advances targetDateLatest to prevent double-counting
- DriftNotification z-index 105 (amber glass toast, 5s auto-dismiss)
- Adaptive fog density: baseDensity + min(taskCount * 0.0002, 0.008)
- Bloom tuned: intensity 1.8, threshold 0.15, smoothing 0.015
- hardDeadline amber pulse takes priority over needsRefinement blue pulse on cards
- Deadline ring 1.2-1.35x radius, refinement ring 1.4-1.6x radius for visual separation
- Drift desaturation: graduated lerp from 0.25 base to 0.7 max toward ETHEREAL_TARGET
- handleReschedule same-horizon guard removed to allow Recommit action
- Canvas display:none (not unmount) for list view — preserves WebGL context
- ListView quick complete skips undo window — list view is for fast triage

## Milestone Summary

**Key Decisions:**

- Stack locked: Next.js 16 App Router, R3F 9.5.0, Drizzle 0.45.1, Zustand 5, Railway Postgres
- Canvas must use `dynamic(ssr: false)` — not just `"use client"` — or the build crashes
- Zustand store must never be imported by Server Components — hard module boundary required
- `horizon` is NOT a DB column — computed client-side from `targetDate` + `now` every render
- Railway Postgres over Neon — simpler setup, persistent Node server, one dashboard
- three@0.170.0 pinned via npm overrides to prevent duplicate bundles
- Boundary values (1, 7, 30, 90, 365 days) inclusive to nearer horizon (<=)
- Past/overdue dates clamp to 'immediate' horizon
- Vitest for testing with TDD red-green-refactor workflow
- cuid2 text primary keys via $defaultFn in Drizzle schema
- API error shape: `{ error: string, code: string }`
- zodOutputFormat with messages.parse() for guaranteed schema-compliant AI extraction
- RSC on-load drift check chosen over cron/client-mount (simplest approach)
- targetDateLatest advanced by window duration (or 7-day minimum) to prevent double-counting

**Issues Resolved:**

- Drift increment trigger ambiguity: RSC on-load check chosen (simplest)
- Haiku JSON reliability: zodOutputFormat with messages.parse() (GA in SDK 0.78.0)
- page.tsx status filter gap: confirmed `WHERE status = 'active'` in final code
- Phase 7 horizon filter UI gap: fixed during audit (26 lines added to ListView)

**Technical Debt Incurred:**

- `useTasksByHorizon` exported from task-store.tsx but never consumed (dead code, harmless)
- No automated E2E tests — all verification via code inspection + manual testing
